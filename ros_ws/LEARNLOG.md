# LEARNLOG

## 相机驱动 + SHM 优化

### 关键教训

#### 1. 分清"每帧延迟"和"帧率瓶颈"
- **错误**：认为端到端 150-200ms 跑不到 20 FPS
- **真相**：帧率取决于**每帧处理时间能否 < 50ms**，不是总延迟
- 相机 DMA + Convert 是每帧必须的固定开销，ROS 拷贝和 SHM 拷贝才是决定能否跑满 20 FPS 的可优化项

#### 2. 优化要抓大头
- **误导**：纠结于 mmap/munmap 每帧开销（~0.4-2ms）
- **真正大头**：ROS 序列化+反序列化（~30ms）、toImageMsg 60MB memcpy（~15ms）
- **教训**：先量化瓶颈再动手，不要凭直觉优化

#### 3. SHM 跨进程设计
- **错误**：把 `shm_fd` 放进 `imageSHM` 共享内存结构体
- **原因**：fd 是进程私有，跨进程无效
- **正确做法**：各进程自己 `shm_open` + `mmap` 获得自己的 fd 和指针

#### 4. ROS 发布去留反复
- **错误**：砍掉 → 恢复 → 又砍掉，消耗了不必要的时间
- **教训**：做架构决策前先确认所有消费方的数据路径，推流走 ZMQ 还是 ROS 要提前定好

#### 5. 不需要额外 API
- **错误**：试图加 `read_image_to(void* dst)` 新函数
- **正确做法**：直接改 `read_image_with_timestamp(void* dst)`，原本的无参版本没有其他调用者
- **教训**：改现有签名比加新函数更干净

#### 6. SDK 接口能力了解不足
- **错误**：没意识到 `MV_CC_ConvertPixelType` 的 `pDstBuffer` 可以指向任意地址
- **后果**：白做了一次内部双缓冲 → SHM 的 memcpy
- **教训**：熟悉底层 SDK 接口能力，避免多余拷贝

#### 7. 采集线程和推理线程职责混淆
- **错误**：把推理的 50ms 也算进采集线程的每帧处理时间
- **正确**：采集线程只管 Convert + 写 SHM（~15ms），推理是独立的流水线
- **教训**：流水线设计下各阶段独立计时
